## Общие правила, ограничения и антипаттерны

## Quick rules

- Не нарушай FSD и зоны импортов (см. [architecture.md](./architecture.md)).
- Не меняй публичные экспорты без необходимости (см. [public-api.md](./public-api.md)).
- Без запроса — без широких рефакторингов.
- Новые зависимости/архитектурные развороты — запрещены без согласования.

## Как использовать AI

- Если задача неоднозначна — сначала найди “source of truth” в существующем коде и повтори паттерн.
- Если задача может затронуть много файлов — держи изменения минимальными и локальными.

### Изменение существующего кода

- Сохранять границы FSD (см. [architecture.md](./architecture.md)).
- Не менять публичные экспорты без необходимости (импорты по всему проекту через баррели; см. [public-api.md](./public-api.md)).
- Без явного запроса — **не** проводить широкий рефакторинг; правки минимальные и по месту задачи.

### Правила выбора при неоднозначности

1. Повторять существующий паттерн в репозитории.
2. Соблюдать FSD и ESLint zones.
3. Использовать `tools/` там, где уже есть генератор/плагин (см. [tools.md](./tools.md)).
4. Предпочитать простое решение.

Запрещено без согласования с командой: новые npm-зависимости, смена архитектуры слоёв.

### Область работ

- **Продуктовый код (по умолчанию):** **`src/`**, **`e2e/`**, **`tools/`**.
- **Конфигурация и инфраструктура репозитория** — допустимы, если это прямо следует из задачи (pnpm/npm-скрипты, сборка, линт, типы, Orval, контейнеры, релиз), например: `package.json`, `pnpm-lock.yaml`, `vite.config.ts`, `tsconfig*.json`, `eslint.config.mjs`, `orval.config.*`, `playwright.config.ts`, `dockerfile`, `.releaserc.json`, корневые `.*rc` / ignore-файлы, **`README.md`**, **`.github/workflows/**`**, обновления правил в **`docs/agents/**`**.
- **Вне scope:** backend и развёртывание вне этого репозитория, если задача об этом не говорит.
- **UI kit** (`@new_york_style/project-template-ui`): не копировать компоненты в `shared` без необходимости; использовать импорты из пакета, как в `src/pages/auth/auth.tsx`, `src/features/auth/ui/sign-in/sign-in.tsx`.

### Антипаттерны

- Нарушение импортов между слоями FSD.
- Inline-стили для визуала.
- Дублирование компонентов вместо переиспользования `shared` / существующих фич.
- Размещение бизнес-ориентированных UI-компонентов в `shared`, если они относятся к пользовательскому сценарию (например OTP-сценарий должен жить в `features`).
- Игнорирование `tools/generators/generate-global-scss.ts` при смене токенов или обход согласованного подключения `variables.scss` / `global.scss` (см. [styles.md](./styles.md)).
- Новые модули без `index.ts` там, где принято публичное API.
- E2E с длинными цепочками селекторов в `.spec.ts` вместо page objects.
- Магические строки данных в тестах вместо общих фикстур/моков.

## Чек-лист

- [ ] Изменения минимальны и по месту.
- [ ] Нет новых зависимостей/архитектурных перестроек.
- [ ] Нет нарушений зон импортов и публичного API.
